# 每週知識庫維護節奏

來源：[[原始資料 03 - 個人知識庫工作流靈感筆記]]

## 核心概念

個人知識庫不需要每天深度整理。更實用的節奏是：平日低摩擦收集，週末集中整理，週日複盤回流。

這樣能避免兩個問題：

- 每天整理太重，導致系統很快放棄。
- 只收集不整理，導致 Inbox 變成新的雜物堆。

## 每週節奏

### 週一到週五：低摩擦收集

只做一件事：把有價值或可能有價值的資料丟進 `00 Inbox 原始資料`。

可以收集：

- 文章
- 影片逐字稿
- 靈感片段
- 專案資料
- 會議紀錄
- 提示詞
- 對話紀錄

這個階段不追求分類精準，重點是降低收集成本。

### 週六：集中整理

請 Codex 協助整理 Inbox。

整理目標：

- 刪掉或封存低價值資料
- 抽出值得沉澱的概念
- 建立或更新 Wiki 筆記
- 找出可以變成 Skill 的流程
- 標記可以變成 Output 的題材

### 週日：複盤回流

檢查這週的輸入、知識、方法與輸出。

複盤問題：

- 這週新增了哪些資料？
- 哪些資料真的有用？
- 哪些內容已經變成知識？
- 哪些流程值得變成 Skill？
- 哪些輸出可以回流成新知識？
- 下週最值得追蹤的主題是什麼？

## 推薦工作流

```text
平日收集 -> 週六整理 -> 週日複盤 -> 下週再收集
```

## 可以自動化的部分

適合讓 AI 處理：

- 掃描 Inbox
- 摘要資料
- 初步分類
- 建議去向
- 生成複盤草稿
- 找出可沉澱的 Skill

需要人工判斷：

- 哪些資料真的重要
- 哪些主題值得長期追蹤
- 哪些輸出符合自己的目標
- 哪些內容應該刪除或封存

## 可行動結論

每週至少完成一次小循環：

1. 清一次 Inbox。
2. 新增或更新至少一篇 Wiki。
3. 沉澱至少一個 Skill 或改進一個既有模板。
4. 寫一份週複盤。
