# 01 開工前｜不打地基就動工 = 全部重做

## 必須先釐清的事（30 分鐘訪談）

### A. 品牌身分

| 問 | 為什麼 |
| --- | --- |
| 公司**登記名稱**是什麼？統編？設立日？ | Footer、JSON-LD legalName 法定資訊 |
| 老闆**個人年資** vs **公司年資**差多少？ | 公平交易法：不能把個人年資算成公司 |
| **對外稱呼**用哪個？登記名 vs 招牌名？ | 影響全站品牌排序 |
| 有沒有**SEO 別名**（行業常見搜尋詞）？ | 在 alternateName 標保 SEO |

> 金智案例：登記名「智廷工程行」、招牌名「金智消防」、SEO 用「金智」。一開始把「金智」當主名，後來客戶堅持以登記名為主 → 全站要改一輪。**提前問可省一個 PR。**

### B. 客戶資料邊界

| 問 | 為什麼 |
| --- | --- |
| 過去案例**可揭露多深**？業主姓名？社區名？地點？ | 案例頁要寫，否則 SEO 失力 |
| 有沒有**保密案**？哪些可以提、哪些連存在都不能說？ | 寫稿時去識別化的尺度 |
| 競爭對手能不能在文章中**指名**？ | 妨害名譽風險 |
| 服務範圍**地區關鍵字**（縣市區）？ | SEO 地區詞 cluster |

### C. 主要客群（不超過 2 種）

| 客群 | 痛點 | 觸發行動 | 想要的文件 |
| --- | --- | --- | --- |
| A 類 | 收到罰單、缺什麼 | 電話 / LINE 急件 | 報價單 / 處理時程 |
| B 類 | 年度作業、預算規劃 | 報價單、開會用 | 正式合約 |

> 多於 2 種客群 = 沒有客群。寧可第二版再加，第一版只服務 1–2 種。

### D. 對外聯絡單一來源

- **電話**：1 支對外（避免分散）
- **Email**：1 個（避免訊息散在多個 inbox）
- **LINE**：1 個官方帳號（避免老闆 / 員工個人 LINE 混雜）

> 多通路 = 內部會吵架。先單線、有量再分。

### E. 隱私底線（不可違反）

- 住商混合的住家地址 → **不要放 Google Map**
- 老闆個人手機 → 寫王小姐 / 業務代表，不寫負責人本人
- 員工本名 → 預設不放（除非他們同意）

## 必出的 3 份文件

| 文件 | 用途 | 範本 |
| --- | --- | --- |
| **SPEC.md** | 產品需求 + 系統設計 + 任務拆解三合一 | 參考 [[金智官網]] / SPEC.md |
| **DESIGN_BRIEF.md** | 給設計師（或 AI）的視覺指示 | 同上 |
| **CMS schema** | Notion DB 欄位定義 | 見 [[03 內容架構]] |

**不寫這 3 份就開工的代價：**做到一半才發現「啊原來年資不能寫公司年資」「啊原來不能放 Google Map」「啊原來客戶不准提名」… 整段內容要重寫。

## 避雷三條（給整個專案的紅線）

1. **法定資訊絕對正確**：統編、地址、設立日。錯一個字 Footer 都要重做
2. **年資切割**：個人 vs 公司絕不混用
3. **競品 / 客戶名**：寫稿先預設不提，需要提時才白名單

## 30 分鐘問完 → 動工前的最後檢查

- [ ] 已知公司全名 + 統編 + 地址 + 設立日 + 負責人
- [ ] 已知主品牌 vs 副品牌（SEO）順序
- [ ] 已知 1–2 種主要客群
- [ ] 已知對外電話 / Email / LINE 各 1
- [ ] 已知**不能寫**的東西（隱私 / 競品 / 客戶名）
- [ ] 已知第一批要做的內容類型：服務頁 / 案例頁 / 知識文章

打勾完整再進 [[02 技術棧]]。
