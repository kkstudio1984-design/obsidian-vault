# 03 內容架構｜Notion DB schema + 三種 page type

## 一個 DB 管全部內容

不要建 3 個 DB（服務 / 案例 / 知識），用**一個 DB + contentType property** 分流。

### 為什麼

- 跨類別查詢方便（最新 6 篇、按客群篩）
- schema 改一次到位
- 編輯者腦袋一個地方
- 程式端 fetch 一次拿全部

### Notion DB schema（最小可用）

```
DB: SEO 文章管理

必要欄位:
- 文章標題      (title)
- Slug          (rich_text)         例：keelung-fire-inspection-guide
- 內容類別      (select)            📝 部落格文章 | 🔧 案例／實戰 | 🛠 服務頁
- 關鍵字類別    (select)            🔴 商業搜尋詞 | 🟡 痛點／流程詞 | 🟠 檢修申報詞 | 🔵 地區詞
- 目標客群      (select)            🟢 業主／店家 | 🏠 社區管委會／主委
- 目標關鍵字    (rich_text)         消防工程公司怎麼挑
- SEO Title     (rich_text)         ≤60 字元
- SEO Description (rich_text)      ≤160 字元
- 封面圖        (files)             1200×630
- 作者          (rich_text)         郭金智｜智廷工程行負責人｜20+ 年實戰
- 上稿日期      (date)              ISO 8601
- 字數          (number)
- 狀態          (select)            待撰寫 | 撰寫中 | 草稿 | 待審 | 已上稿
- Tags          (multi_select)      自由標籤，給延伸閱讀演算法用
- 支柱文章      (relation → 自己)   Topic Cluster 連結
- 🚀 立刻發布   (button)            webhook 觸發 ISR

注意:
- 狀態用 select 而非 status (Notion API 不支援建 status)
- emoji 寫死在 select 值，前端 grep 用
```

## 三種 page type 的差別

| 屬性 | 服務頁 🛠 | 案例頁 🔧 | 知識文章 📝 |
| --- | --- | --- | --- |
| 目的 | 「我們提供什麼」 | 「我們做過什麼」 | 「我們懂什麼」 |
| URL | `/services/<slug>` | `/cases/<slug>` | `/knowledge/<slug>` |
| 字數 | 600–800 字 | 750–850 字 | 1500–2000 字 |
| 結構 | 適用情境 + 流程 + 費用 + 完工交付 + FAQ | 背景 + 處理 + 結果 + 給類似業主的建議 | 完整法規 + 流程 + Q&A + 延伸閱讀 |
| SEO 力道 | 中（轉換用） | 中（信任用） | 強（流量用） |
| 必要欄位 | 服務名 / 範圍 / 價格區間 | 時間 / 地點 / 業主類型 / 結果 | 完整關鍵字策略 |
| 改寫風險 | 低 | **高**（去識別化） | 中 |

### 服務頁範本結構

```markdown
## 適用情境
- 場景 1
- 場景 2
- 場景 3
- 場景 4

## 我們怎麼處理（流程）
1. 第一步
2. 第二步
3. 第三步

## 費用怎麼估
- 因子 1
- 因子 2

## 完工後你會拿到
- 交付物 1
- 交付物 2

## 常見問題
**Q1**
A1

**Q2**
A2

---
> CTA：撥打 XXX 找 YYY，免費評估
```

### 案例頁範本結構（高去識別化）

```markdown
## 案件背景
（時間 + 地區 + 業主類型 + 規模，不含具名）

## 我們怎麼處理
1. step 1
2. step 2

## 技術重點 / 為什麼這樣做
（差異化技術點）

## 結果與後續
- 客觀結果
- 客戶反饋

## 給類似情境業主的建議
- 通用 takeaway

---
> CTA
```

### 知識文章範本結構（SEO 主力）

```markdown
## 為什麼這件事重要

## 法規 / 標準說明
（含表格）

## 實務流程

## 費用 / 時程 / 規範

## 完工後 / 應有產出

## 常見問題

## 給某類客群的建議

---
> CTA
```

## Topic Cluster 結構

```
🔴 商業搜尋詞      Pillar: 「消防工程公司怎麼挑」
   ├── 子文: 消防工程費用解析
   ├── 子文: 廠房消防工程
   └── 子文: 餐廳消防工程

🟡 痛點/流程詞     Pillar: 「使用執照卡消防怎麼辦」
   ├── 子文: 消防圖審退件
   └── 子文: 消防送審流程

🟠 檢修申報詞      Pillar: 「集合住宅消防檢修申報」
   └── ...

🔵 地區詞          Pillar: 「基隆消防工程實戰」
   ├── 子文: 台北消防工程
   ├── 子文: 新北消防工程
   └── 子文: 桃園消防工程
```

每個 cluster 6 篇起：1 支柱 + 5 子。30 篇 = 4 cluster × 7-8 篇 + 一些零散。

## fallback chain（網站讀 Notion 的安全機制）

```
1. Notion 封面圖（業主上傳的）
2. 服務/案例/知識 → /images/<type>/<slug>.png（程式端 fallback）
3. 都沒有 → /images/og-default.jpg
```

實作見 `lib/notion.ts` `mapPageToArticle()`。

## 延伸閱讀演算法（給文章詳情頁底部）

三層 tier 找 3 篇相關：

```
Tier 1: 同 Tags 交集 ≥ 2 篇
Tier 2: 同 關鍵字類別
Tier 3: 同 內容類別
```

實作見 `lib/related.ts`。
