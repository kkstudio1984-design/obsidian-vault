# 反面教材｜這些雷我們踩過

每條都是真實事件 → 之後可避。

## 1. 「主品牌」沒在開工前敲定 → 整改一輪

**情境：** 開工時用「金智消防」當主名做了 50+ 個檔案的 metadata / JSON-LD / 文案。一週後客戶說「我們堅持登記名稱『智廷工程行』為主」→ 13 處 code references + Footer + Logo + JSON-LD 全改。

**教訓：** 開工前 30 分鐘的訪談，**主品牌排序務必白紙黑字確認**。不要用「先做做看再說」。

## 2. 域名沒先查就接 Vercel

**情境：** 假設 `kingzhi.com` 是客戶的 → 開始規劃 NEXT_PUBLIC_SITE_URL → 後來 whois 一查發現 2012 年就被中國某公司搶註，跟客戶 2023 年成立的公司無關。

**教訓：** **開工前 5 分鐘做的事：** whois 客戶想要的域名。如果不是客戶持有 → 看價格買 / 用替代域名（`.tw` / `.com.tw`）。

## 3. 案例頁寫得太具體 → 法律風險

**情境：** 客戶提供「北部 X 物流大型儲運廠」案例，含「前廠商作假」故事。第一版寫稿時備註標了真實品牌名暗示 → 差點進正式稿。若上線會踩**妨害名譽**紅線。

**教訓：** 案例稿件**預設去識別化**到「北部某產業園區」等級。具名揭露需業主+老闆雙重書面同意。

## 4. Notion multi-source DB API 卡半天

**情境：** 客戶有新 DB（用新版 Notion multi-source 結構），整合連到父頁面但**無法 query rows**。錯誤訊息「does not contain any data sources accessible by this API bot」極不直觀。

**教訓：** Notion 2025-09 改架構，新 DB 要：
- 用 `Notion-Version: 2025-09-03` API
- 連 connection 在**內嵌 DB 本身**（不繼承父頁）
- Query 用 `/v1/data_sources/{ds_id}/query` 不是 databases endpoint

## 5. Vercel Hobby + private repo 擋 commit

**情境：** 全域 git config 用 email A，但 Vercel 用 email B → push 後 Vercel build 直接 fail，error 訊息超不直觀。

**教訓：** Hobby 方案有限制 → 該專案 repo 跑 `git config user.email <vercel-email>` 覆蓋全域。

## 6. og-default.jpg 1.3MB 沒壓縮

**情境：** AI 產的 og-default 是 1.3MB PNG → 每篇文章社群分享 fallback 都載這張 → 拖慢 OG 抓取 → 影響社群預覽 CTR。

**教訓：** **OG 圖必壓縮**到 ≤ 300KB。用 sharp / TinyPNG / squoosh.app 都行。

## 7. Notion 「狀態」用 select 不是 status

**情境：** SPEC 寫 status 型別，但 Notion API **不支援用 API 建 status property**（只能 UI 建）→ 程式跑出來抓不到。

**教訓：** 跨機器跨人協作的 Notion DB → **狀態欄位用 select**，避免相容性問題。

## 8. 一個 PR 包太多事

**情境：** PR #9 原本只是「安全升級 + SEO metadata」，後來臨時加 8 張封面圖一起合 → 17MB 二進位 + 程式 diff 混在一起，PR title 跟內容 diverge。

**教訓：** **一個 PR 一個動詞**。臨時想加事 → 開新 PR。

## 9. 圖片資源沒中央備份

**情境：** AI 圖直接從下載資料夾 cp 到 `public/images/`，原檔放 Downloads。下載資料夾整理時誤刪 → 想改尺寸要重新產。

**教訓：** 中央 assets 資料夾（repo 外、名為 `_orig` 後綴），原檔備份留著。

## 10. 想一次規劃完美的 Obsidian / Notion folder 結構

**情境：** 花一週時間想「PARA vs 卡片盒」vs「Zettelkasten」vs「LATCH」哪個好 → 一篇筆記都沒寫。

**教訓：** **先動起來再優化**。初版用 PARA，一個月後依實際使用情況再調整資料夾。

## 11. og:image 用 1200×630 vs 文章卡用 16:9 比例不對齊

**情境：** OG 標準 1.91:1 (1200×630) 但 ArticleCard 用 16:9 (1.78:1) → 圖會 crop 上下幾%。

**教訓：** 設計圖時**主物件留中央**，上下邊緣別放重點。1.91:1 vs 16:9 差距 3-4%，可接受。

## 12. Search Console 「URL 前置字元」忘加結尾 /

**情境：** 輸入 `https://kingzhi-web.vercel.app` → Google 跳紅字「請輸入完整 URL 前置字元」→ 卡住。

**教訓：** 結尾必須 `/`，例 `https://kingzhi-web.vercel.app/`。Google 沒告訴你規則。
