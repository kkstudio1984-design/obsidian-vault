# 90 分鐘優化版：從課程會議記錄到 AI 工作流與 LLM Wiki

## 版本定位

這份 90 分鐘版本，是根據 2026-05-29 工研院 Day 1 上課會議記錄濃縮而成，並整合「20 分鐘 Web Clipper 到 LLM Wiki」講座內容。

這版不是全天課程回放，而是給已經上過課、或想快速複習的人，用 90 分鐘抓住 Day 1 最重要的主線：

```text
AI 使用等級 -> 痛點盤點 -> 流程拆解 -> 知識收集 -> PRD 草稿 -> 下週 MVP 準備
```

## 一句話課程承諾

> 90 分鐘後，學員會知道如何把自己的營運痛點，透過 AI 對話整理成需求說明書，並把過程中的資料沉澱進 Notion / Obsidian / LLM Wiki，成為後續可迭代的知識系統。

## 適合對象

- 參加過 5/29 Day 1，但想快速複習的人。
- 還沒完成 PRD 草稿的人。
- 想知道 Codex、Notion、Obsidian、LLM Wiki 怎麼分工的人。
- 想把課堂筆記、會議紀錄、網頁資料整理成可用知識的人。

## 90 分鐘主線

```text
先定位置：我現在 AI 使用到哪一級？
再找痛點：我最想解決哪個營運卡點？
再拆流程：誰負責？看什麼資料？做什麼判斷？
再收資料：用 Notion / Obsidian / Web Clipper / LLM Wiki 留下來源。
再寫 PRD：讓 AI 擔任產品經理，問我 6 輪問題。
最後定作業：下週帶真實資料，做 MVP。
```

## 0:00-8:00 開場與 AI 使用等級

### 講師目標

先讓學員知道：這堂 90 分鐘不是工具展示，而是把 Day 1 重新濃縮成「可以回家做」的路徑。

### 講師台詞

> 昨天我們從早到晚跑了很多工具：Codex、Notion、LLM Wiki、Google AI Studio、Notebook LM，也做了痛點盤點和 PRD 草稿。今天這個 90 分鐘版本，我不會重播所有細節，而是幫大家整理成一條可以照做的路線。
>
> 你只要記得一件事：AI 不是只拿來聊天，而是拿來幫你把痛點變成流程，把流程變成需求說明書，再把資料沉澱成知識系統。

### AI 使用等級快速定位

用 1 分鐘請學員自評：

- Level 1：知道 AI 可以做事，但還沒真正用在工作。
- Level 2：會用提示詞問問題。
- Level 3：會拆任務，知道哪些工作可交給 AI。
- Level 4：有很多工具，但不知道什麼時候用哪個。
- Level 5：有自己的工作流程和 SOP。
- Level 6：開始設計系統，讓 AI 參與流程。
- Level 7：有完整知識管理策略，不被單一工具綁架。

### 轉場句

> 多數人昨天大概在 Level 2 到 Level 3。這 90 分鐘，我們要把大家往 Level 5 和 Level 6 推一小步：不只是會問 AI，而是會讓 AI 幫你整理流程與需求。

## 8:00-18:00 Day 1 核心回顧

### 昨天全天濃縮成 5 件事

1. 認識 Codex：AI 不只是大腦，也可以成為雙手。
2. 找出痛點：不要一開始就做系統，先知道卡在哪裡。
3. 拆解流程：誰負責、看什麼資料、做什麼判斷。
4. 建立知識系統：Notion / Obsidian / LLM Wiki 分工。
5. 產出 PRD：把痛點變成工程師看得懂的需求說明書。

### 講師台詞

> 昨天不是工具課，而是轉換思維的課。以前我們可能會說：「我想做一個庫存系統。」但 AI 時代更好的問法是：「我的庫存流程在哪個判斷點卡住？這個判斷需要哪些資料？哪一段可以交給 AI 協助？」

### 工具分工簡表

| 工具 | 定位 | 什麼時候用 |
|---|---|---|
| ChatGPT / Gemini / Claude | 大腦 | 問答、分析、生成內容 |
| Codex | 雙手 | 操作檔案、同步資料、建立文件 |
| Notion | 對外資料庫 / 專案管理 | 報告、任務、協作、資料庫 |
| Obsidian | 個人知識庫 | 筆記、複盤、學習沉澱 |
| LLM Wiki | AI 維護的知識系統 | 把來源整理成可查、可連結的知識 |
| Notebook LM | 大量來源萃取 | PDF、YouTube、文件摘要與問答 |

## 18:00-32:00 痛點盤點與 MOT

### 講師目標

讓學員從「我想做系統」退一步，先寫出真正痛點。

### 昨天常見痛點類型

- 店員靠記憶處理折扣規則。
- 顧客停下但沒被接住，銷售時機流失。
- 商品效期、選價、報價追蹤缺少提醒。
- 庫存盤點和耗材預測需要人工判斷。
- 客戶分類不清，低毛利客戶消耗人力。
- 前置資訊不標準，老闆一直被迫人工追問。

### 痛點句模板

```text
當「某個情境」發生時，
因為「缺少某個資料 / 判斷 / 流程」，
導致「某個人」必須人工處理，
最後造成「時間、成本、客戶體驗或營收」上的損失。
```

### 現場練習 5 分鐘

請學員寫一句自己的痛點句：

```text
當 __________ 發生時，
因為 __________，
導致 __________，
最後造成 __________。
```

### 講師示範

```text
當每日打烊後要檢查原物料庫存時，
因為 iChef 銷售報表和 Excel 庫存表沒有自動連動，
導致老闆必須人工比對每個品項與配方，
最後造成盤點耗時、補貨延遲與耗損難以追蹤。
```

## 32:00-50:00 流程拆解：誰看什麼做什麼判斷

### 講師目標

把痛點拆成 AI 可以介入的流程，而不是直接跳去做工具。

### 三問框架

```text
1. 誰負責？
2. 要看哪些資料？
3. 要做什麼判斷？
```

### 四格流程

```text
進貨 -> 銷售 -> 庫存 -> 報表 / 決策
```

### 現場練習 8 分鐘

請學員把剛剛的痛點放進四格流程：

| 流程 | 現在怎麼做 | 卡點 | AI 可介入 |
|---|---|---|---|
| 進貨 |  |  |  |
| 銷售 |  |  |  |
| 庫存 |  |  |  |
| 報表 / 決策 |  |  |  |

### 講師台詞

> AI 最怕你只說：「幫我做一個系統。」它真正需要的是：誰在什麼時間點，看什麼資料，做什麼判斷。這三件事越清楚，AI 越能幫你寫出接近可開發的需求說明書。

### 小結

```text
痛點不是需求。
痛點 + 流程 + 資料 + 判斷，才會變成需求。
```

## 50:00-68:00 整合 20 分鐘講座：Web Clipper 到 LLM Wiki

### 這段的定位

這裡放入 20 分鐘講座的核心，但壓縮成 18 分鐘。它的任務不是完整教 Web Clipper，而是回答：

> 課堂中產生的資料、網頁、會議記錄、提示詞，要怎麼被保存下來，變成下次能用的知識？

### 50:00-53:00 心智模型

```text
網頁 / 會議記錄 / Prompt / 課堂討論
        ↓
00 Inbox 原始資料
        ↓
10 Wiki 知識庫
        ↓
20 Skills 方法庫
        ↓
30 Outputs 輸出
        ↓
40 Reviews 複盤
```

講師台詞：

> 昨天我們產生了很多資料：痛點牆、會議記錄、PRD 草稿、提示詞、工具設定。這些如果不收進系統，課後就會散掉。Web Clipper 和 Inbox 的價值，就是先把資料留下來，之後再讓 Codex 幫你整理。

### 53:00-58:00 Web Clipper 最小設定

只講三個欄位：

```text
Vault：Obsidian-vault
Note location：00 Inbox 原始資料/Web Clipper
Note name：{{date}} - {{title}}
```

提醒：

> Vault 要填 vault 名稱，不是 Windows 路徑。不要填 `D:\Obsidian-vault`。

### 58:00-63:00 剪藏後怎麼整理

給 Codex 的指令：

```text
請整理「00 Inbox 原始資料/Web Clipper」裡最新剪藏的資料。

請判斷：
1. 它的核心主題是什麼？
2. 是否值得整理到 10 Wiki 知識庫？
3. 是否能沉澱成 20 Skills 方法庫？
4. 是否能變成 30 Outputs 輸出？
5. 請標記整理狀態。
```

### 63:00-68:00 三種去向

| 去向 | 何時使用 | 範例 |
|---|---|---|
| Wiki | 有長期概念 | LLM Wiki 自生長知識庫 |
| Skill | 有可重複流程 | Inbox 資料整理 Skill |
| Output | 可交付成果 | 90 分鐘講座稿、PRD 草稿 |

講師金句：

> 能解釋世界的進 Wiki，能重複操作的進 Skill，能對外交付的進 Output。

## 68:00-82:00 PRD 草稿生成

### 講師目標

讓學員知道昨天最後產出的核心作業是 PRD，不是單純聊天紀錄。

### PRD 是什麼

PRD 是產品需求說明書。它的功能是把人話轉成工程師、AI 或系統設計者看得懂的規格。

### PRD 生成流程

```text
痛點句 -> 流程拆解 -> MVP 範圍 -> 產品經理 6 輪追問 -> PRD 草稿
```

### 現場指令 1：先找 MVP

```text
我想解決以下痛點：
「貼上你的痛點句」

請你擔任零售 / 批發 / 進銷存報系統顧問，幫我判斷這個痛點最適合先做哪一個 MVP。

請不要一開始設計完整 ERP，只要找出第一版最小可行功能。
```

### 現場指令 2：產品經理 6 輪追問

```text
請你擔任我的產品開發經理。

我要把這個痛點整理成可以交給工程師或 AI 開發工具使用的 PRD。

請一次只問我 1 到 2 個問題，總共進行 6 輪對話。
每一輪都要幫我補齊需求說明書中容易漏掉的角度。

6 輪結束後，請幫我整理成一份 PRD 草稿。
```

### PRD 草稿至少包含

- 背景與問題
- 使用者角色
- MVP 範圍
- 資料來源
- 核心流程
- 功能需求
- 欄位需求
- 權限需求
- 例外情境
- 驗收標準
- 下一步資料準備

### 講師台詞

> 這裡的重點不是讓 AI 一次寫出完美 PRD，而是讓它變成產品經理，反過來問你問題。你回答得越具體，需求說明書越能真的被開發。

## 82:00-90:00 收尾與回家作業

### 90 分鐘總結

1. 先定位：你現在 AI 使用到哪一級。
2. 再寫痛點：不要一開始就做工具。
3. 再拆流程：誰負責、看什麼、做什麼判斷。
4. 再收知識：Web Clipper / Inbox / LLM Wiki 讓資料不散掉。
5. 最後寫 PRD：把痛點變成能開發的需求。

### 回家作業

請在下次上課前完成：

1. 寫出 1 句痛點句。
2. 畫出 1 張四格流程：進貨、銷售、庫存、報表 / 決策。
3. 用 AI 進行 6 輪產品經理追問。
4. 產出 1 份 PRD 草稿。
5. 收集至少 1 份真實資料，例如 CSV、Excel、報表、截圖或表單。

### 下次上課要帶

- PRD 草稿
- 真實資料
- 目前使用的表單 / Excel / POS 報表
- 一個你願意先做 MVP 的流程

### 最後金句

> AI 不會自動理解你的公司，但它可以幫你把你說得清楚的流程，變成可以執行的系統。

## 講師備忘：這版和 20 分鐘講座的關係

20 分鐘講座只教：

```text
Web Clipper -> Inbox -> Codex -> Wiki / Skill / Output
```

90 分鐘版本要教：

```text
痛點 -> 流程 -> 資料 -> 知識系統 -> PRD -> MVP
```

所以 20 分鐘內容不要獨立講太久，只放在 50:00-68:00，作為「資料如何被沉澱」的中段示範。

## 講師備忘：現場不要展開的內容

90 分鐘內不要深入：

- 完整 Codex 安裝排錯
- LLM Wiki 技術原理
- Notion 資料庫教學
- Notebook LM 全功能
- 自動化排程
- 完整系統開發

這些都可以留到後續課程。
